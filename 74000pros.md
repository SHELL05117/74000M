## 4. 全局硬不变量

- 全工程只有 `OutputService` 持有可写 `DriveIO` 并写驱动电机。
- 每拍只采集一份传感器快照；消费者只读同一份不可变状态。
- 核心统一 SI：m、rad、s、m/s、rad/s、V、A、°C；HAL 才处理 degree、centidegree、RPM、mV。
- 每个跨任务帧含单调时间戳、sequence、mode epoch；请求和输出分别有 TTL。
- 模式切换采用 break-before-make：撤销旧 owner、清有记忆状态、先停止，再授权新 owner。
- 所有请求都经过 `DriveRequest -> Scheduler/Arbiter -> Drivetrain -> SafetyGate -> ActuatorFrame -> OutputService`。
- 安全门位于控制器之后、执行器之前；Disabled、断控、陈旧、非有限数和锁存停止不能被覆盖。
- 控制环不做动态分配、文件/SD I/O、长字符串格式化、屏幕刷新、传感器校准等待或无界锁等待。
- 位姿、航向和传感器量必须携带质量与年龄；IMU-only 不得标为完整平面位姿有效。
- 所有参数绑定 robot ID、schema、revision、数据集和适用工况；示例参数不能直接成为比赛参数。

## 6. 最快交付方案：离线并行、能力锁闭、真机集中验收

### 6.1 总原则

目标是在机器人机械搭建期间完成尽可能多的可编译、可测试底层代码，并把真机时间集中用于事实核验、标定和批准，而不是现场补架构。

- 文档门禁、安全不变量和唯一输出链路始终强制执行。
- 板块编号表示功能与验收范围，不再表示必须串行完成代码。
- 软件接口依赖仍需满足；硬件验收缺失可用 Fake、仿真和 capability lock 隔离。
- 所有预设参数必须携带来源与验证状态；示例值、CAD值和估计值不得伪装成比赛参数。
- 未通过硬件输出解锁前，PROS 目标即使编译成功也必须默认停车。
- 未通过底层冻结门和速度闭环验收前，`autonomous_chassis_velocity`、自动运动和路线 capability 必须保持 `false`；有效自动选择只能回退 `DoNothing`。

### 6.2 离线阶段 A：契约与工程骨架

优先完成板块 0–2 的可离线部分：

1. 冻结 PROS+C++17、坐标系、robot ID、schema、稳定 ID 和首版功能范围。
2. 对未知端口、反向、几何和传感器布局建立显式 `Unverified/PendingCalibration` 状态，配置校验默认拒绝输出解锁。
3. 建立 V5 与 PC 双目标、测试框架、FakeClock/Fake IO、静态架构检查和 CI。
4. 完成核心单位、帧头、质量、故障、状态、请求、执行器帧、快照和注册表类型。

交付物：可在 PC 测试的核心库、可编译的 PROS 空壳、默认锁闭的配置和接口契约。

### 6.3 离线阶段 B：四条开发泳道并行

各泳道内部按代码依赖推进；泳道之间以冻结接口并行开发。

1. **运行时与安全泳道**：板块 3、4、6、10、11。实现 PROS HAL 适配、ModeManager、ControlLoop、唯一 OutputService、双 TTL、Scheduler/Arbiter、运动学分配、安全门和故障状态机。硬件写路径通过 Fake/spy 验证，真实输出保持锁闭。
2. **状态与测试泳道**：板块 5、7、8。实现日志环、FakeDriveIO、重放、传感器校验、速度估计、SE(2)、多种里程计和故障注入；用解析数据、合成噪声、异常样本和现有仿真基线验证。
3. **手动与 HMI 泳道**：板块 9、13。实现控制器快照、输入曲线、Slew、曲率/Quick Turn、HmiModel/Event、注册表、事务、回滚和持久化；真实驾驶参数保留为 Draft，路线只允许 `DoNothing` 被批准。
4. **自动算法泳道**：板块 15–21。实现平台无关 PID、前馈接口、运动规划、非阻塞命令、速度请求、轨迹生成/跟踪、路线编排和失败策略；全部通过 PC/Fake/仿真测试，但所有自动 capability 默认关闭，不生成可绕过安全门的硬件输出。

板块 12、14、22 可先完成工具、数据格式、测试用例、报告模板和离线步骤；涉及机械、HIL、场地和比赛批准的结论保留 `NOT TESTED`。

### 6.4 离线阶段 C：全链 Fake 闭环集成

在硬件到位前完成：

1. 用 FakeClock 驱动确定性 10 ms 名义循环并覆盖可变 `dt`、时间回退、超期和任务冻结。
2. 贯通 `DriveRequest -> Scheduler/Arbiter -> Drivetrain -> SafetyGate -> ActuatorFrame -> OutputService -> FakeDriveIO`。
3. 注入 Disabled、断控、旧 epoch、旧 lease、请求 TTL、输出 TTL、NaN/Inf、端口错误、IMU失败、单电机异常、热/流/卡死/低电候选数据。
4. 完成直线、旋转、圆弧、多圈、重置、路径连续性、约束、取消、timeout、stall 和状态质量测试。
5. 运行唯一写者、核心无 PROS 头、无 `12/Vbattery`、无阻塞自动循环和无直接电机访问的静态审计。
6. 生成一份未验收矩阵，逐项标明 `PCValidated/SimValidated/NOT TESTED` 和需要的真机证据。

交付物：除真实物理参数和现场性能外，完整可运行的离线底层与自动控制软件；默认输出和自动能力仍锁闭。

### 6.5 硬件到位后的集中验收顺序

严格按 `docs/HARDWARE_COMMISSIONING.md` 执行：

1. 机械、接线、设备身份、方向、单位和 brake 路径；满足后才解锁受限测试输出。
2. 真机任务周期、采集跨度、请求/输出 TTL 和 UI/日志压力；冻结运行时参数。
3. 编码器、IMU、tracking wheel 方向、质量、恢复和里程计几何；完成直线、旋转和闭合校准。
4. 基础手动、曲率、Quick Turn、Slew、断控和安全矩阵；完成热、电流、卡死、滑移和低电基线。
5. SysId、分侧前馈、速度闭环和底层冻结门；只有通过后才开启 `autonomous_chassis_velocity`。
6. 低速自动原语、点到点、圆弧、轨迹、整条路线和故障注入逐级验收；每一级单独解锁 capability。
7. 完成 HIL、FieldValidated、CompetitionApproved、配置哈希、回滚版本和比赛冻结。

### 6.6 能力解锁硬门

允许跨板块写代码不等于允许提前驱动：

- `hardware_output`：端口、方向、单位、急停和 brake 路径未真机通过时为 `false`。
- `driver_control`：断控、模式、TTL、有限数和基础停止矩阵未通过时为 `false`。
- `pose_good`：几何、方向和对应传感器组合未校准时不得发布完整 `Good`。
- `autonomous_chassis_velocity`：速度闭环、状态质量门和底层冻结未通过时为 `false`。
- `autonomous_motion`：非阻塞原语、取消/超时/卡死和低速真机测试未通过时为 `false`。
- `competition_routes`：路线未逐条 FieldValidated/CompetitionApproved 时只能执行 `DoNothing`。
